# Traceability gaps — where `@trace` was deliberately NOT written

Generated by hand from the `trace-annotations` workflow (9 annotator agents +
24 adversarial verifiers, maker ≠ checker), 2026-07-10.

**20 of 34 MVP FRs carry an honest `@trace` annotation. 14 do not.** This file
records why, per requirement. An absent annotation is a documented gap. A false
annotation would be forged evidence — the same class of defect as the ten
back-stamped `review-findings.json` deleted in `improve-PD-3`.

The test applied to every annotation: **if this requirement were silently
broken, would this specific test fail?** "Right module, right function, adjacent
behavior" is not proof.

## Refuted by adversarial verification (5 of 24 claimed annotations, 21%)

These were annotated by a capability agent, then a fresh agent — shown only the
spec and the test, never the annotator's justification — refuted the claim.

| FR | Claimed on | Why the claim failed |
| --- | --- | --- |
| `FR-CALC-01` | `numbering.test.ts` | The tests exercise two pure functions over a flat `string[]`. Three of five scenarios are unprovable there: **per-supplier scoping** (the function has no supplier concept — a single global counter would pass every assertion), **draft-carries-no-number**, and **cancelled-number-not-reused** (the real risk is the *caller* forgetting to keep cancelled numbers in `existing`; a pure function handed the number cannot detect that). |
| `FR-INPUT-01` | `form-to-render.test.ts` | Asserts only that no `{{TOKEN}}` survives — a template-completeness property. Drop a required input field, or hardcode it, and the test still passes. |
| `FR-BANK-02` | `supplier-profiles.test.ts:214` | The annotated test verifies SWIFT/IBAN **normalization** (uppercase, whitespace strip). Stop persisting `addressUa` and it still passes. *Relocated* — see below. |
| `FR-EXPORT-02` | `download-invoice-html.test.ts` | Feeds a stub `"<html><body>Invoice</body></html>"` and asserts blob MIME type, filename, anchor click. The requirement's defining property — **a self-contained file that opens offline with embedded styles matching the live preview** — is structurally unprovable by this test. |
| `FR-EXPORT-03` | `print-invoice-html.test.ts` | The `210mm × 297mm` is the hidden iframe's **on-screen box** (so browsers don't print a blank page), not the print page size — that lives in the template's `@page { size: A4 }`. Change it to Letter, or overflow the body horizontally, and the test still passes. jsdom does no layout. |

### One relocation

`FR-BANK-02` moved to `supplier-profiles.test.ts` → `"drops invalid records and
keeps valid ones"`. Its `expect(listProfiles()).toEqual([validRecord])` is a
**strict** object comparison across all nine fields. Verified by falsification:
breaking `addressUa` in the storage round-trip fails the test.

## Defect found in the product's own tests

`numbering.test.ts` contained a test named
`"never reuses a cancelled number still present in the register"` whose body was
**behaviourally identical** to the adjacent `"advances the sequence"` test —
`["2026-001","2026-002"] → "2026-003"`. The word *cancelled* existed only in a
comment. This is a forged evidence claim living in a test name. Renamed to
`"skips every number present in the register, whatever its status"`. Assertions
unchanged.

## Traceability defect: `client-directory` owns nothing

`openspec/capability-map.yaml` lists `client-directory` with an empty `owns: []`,
yet `src/lib/storage/clients.test.ts` holds 10 passing tests and
`openspec/specs/client-directory/spec.md` describes real behavior. No `FR-CLIENT-*`
ids exist in `docs/requirements.md`.

Consequence: a shipped capability with tests and a spec sits **outside the FR
chain entirely**. `check-traceability` cannot see it, because it only walks the
requirement ids in `requirements.md`. The capability could regress silently and
no gate would notice.

The annotator agent refused to invent an id, correctly. Numbering these
requirements is a scope decision for the human, not an agent.

## Open process defects touching this work

- **PD-1** — `check-traceability` has no `dropped` status, so `FR-NACE-06` and
  `FR-INPUT-03` are counted as MVP obligations forever.
- **PD-12 (new, unfiled)** — the `@trace` parser matches the annotation
  **anywhere in a file**, including inside prose and negations. While removing
  the five refuted annotations, comments reading `// No @trace FR-EXPORT-02: …`
  were silently parsed as annotations and re-credited all four requirements. The
  same shape bit a test fixture earlier (`tests/gate-fixtures/…/money.test.ts`
  credited the real `FR-CALC-01`). Minimal fix: anchor the pattern to the start
  of a comment — `^\s*(?://|\*)\s*@trace\s+`. Three independent paths to one
  defect means the mechanism invites the mistake.
